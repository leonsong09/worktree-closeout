# closeout Artifact Format

Use this reference when reading the Markdown artifact generated by `scripts/scan_closeout.py`.

## Frontmatter Fields

The artifact frontmatter is YAML and currently uses these fields:

| Field | Meaning |
| --- | --- |
| `skill` | Always `closeout` |
| `date_scope` | Resolved scan date in `YYYY-MM-DD` |
| `scan_scope` | Scanner scope: `repo` or `all` |
| `generated_at` | Local timestamp when the artifact was written |
| `repo_count` | Number of scanned repos in the artifact |
| `worktree_count` | Number of scanned worktree entries |
| `branch_count` | Number of distinct branch labels found |
| `safe_prune_count` | Count of items classified as `safe_prune` |
| `ready_to_merge_count` | Count of items classified as `ready_to_merge` |
| `blocked_count` | Count of items classified as `blocked` |
| `orphaned_count` | Count of items classified as `orphaned` |
| `status` | `needs-closeout` or `empty-scan` |
| `source_sessions_dir` | Session directory used as the evidence source for that date |

## Body Section Order

The Markdown body is rendered in this order:

1. `# closeout`
2. `## 总览`
3. `## 建议收口顺序`
4. `## 按 Repo 分类结果`
5. `## 建议总控 Prompt`
6. `## 建议子 Prompt`

Within `## 按 Repo 分类结果`, each repo section uses:

1. `### Repo: <path>`
2. `- 会话数: <n>`
3. category blocks in this fixed order:
   - `#### safe_prune`
   - `#### ready_to_merge`
   - `#### blocked`
   - `#### orphaned`

Each item is rendered as:

```md
- `<label>` @ `<worktree-path>`
  - <notes>
```

If a category has no items, it renders as `- NONE`.

## Classification Meanings

### `safe_prune`

Merged branch + clean worktree. Good candidate for manual cleanup review. The artifact does **not** authorize automatic deletion.

### `ready_to_merge`

Not yet merged, clean, ahead of base, and backed by recent session evidence. Good candidate to hand off to a branch-finishing flow.

### `blocked`

Something still prevents safe closeout: dirty state, behind-base state, unfinished base-branch work, or another signal that the item is not ready.

### `orphaned`

The current git/session evidence is too weak or incomplete to justify merge/prune confidently. Re-triage first.

## Prompt Section Conventions

### Controller Prompt

`## 建议总控 Prompt` contains a fenced `text` block. It is the top-level orchestration prompt for the whole artifact and should:

- read the artifact for the chosen date/scope
- process items in category order:
  `safe_prune -> ready_to_merge -> blocked -> orphaned`
- keep risky actions manual

### Per-item Prompts

`## 建议子 Prompt` contains one bullet per closeout item:

```md
- `<classification>` :: `<repo>` :: `<label>`
  - <prompt text>
```

Conventions:

- `<classification>` is one of the four closeout categories
- `<repo>` is the absolute repo path
- `<label>` is usually the branch name, or the worktree name when no branch is available
- the indented prompt is a starting instruction for that item, not a license for auto-merge / auto-delete / auto-push

Category prompt intent:

- `safe_prune`: confirm merged state and cleanup readiness
- `ready_to_merge`: run finishing checks and prepare a merge-or-keep recommendation
- `blocked`: identify blocker and propose the smallest next unblock step
- `orphaned`: re-classify from current git state plus recent session clues
